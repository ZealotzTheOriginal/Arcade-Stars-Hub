====================================================
  ARCADE STARS HUB — Resumen de Implementación
====================================================

STACK
-----
- Frontend: Angular 17 (standalone components, signals, effect())
- Backend:  FastAPI + WebSockets + asyncio
- BD:       Firebase Firestore + Firebase Auth
- IA:       DeepSeek API (fallback) + algoritmos propios (principal)
- Deploy:   Render (frontend estático + backend como servicio web)


ARQUITECTURA GENERAL
--------------------
- WS gestionado a nivel raíz (App component), NO por sala.
  Reason: las notificaciones globales (invitaciones, etc.) deben
  funcionar en cualquier página.
- El backend mantiene dos diccionarios en memoria:
    _rooms:    {room_id: dict}    — estado de cada sala
    _presence: {uid: dict}        — usuarios registrados como online
- ConnectionManager tiene dos planos:
    _rooms:  {room_id: {uid: WebSocket}}  — conexiones por sala
    _direct: {uid: WebSocket}             — canal directo por usuario
- Ciclo de sala: waiting → playing → finished → deleted


FUNCIONALIDADES IMPLEMENTADAS
------------------------------

1. AUTENTICACIÓN
   - Firebase Auth (email + contraseña)
   - Rutas protegidas por guard Angular
   - Token verificado en backend con Firebase Admin SDK

2. JUEGOS BASE (5 juegos)
   - Conecta Cuatro  (multijugador + IA)
   - Tres en Raya    (multijugador + IA)
   - Buscaminas      (multijugador por turnos + IA)
   - Pong            (multijugador en tiempo real, WebSocket tick-loop 30 fps)
   - Snake           (multijugador en tiempo real, WebSocket tick-loop)
   - Sistema de puntos: se guardan por juego en Firestore
   - HUD con chat en tiempo real vía WS

3. IA POR JUEGO
   - Tres en Raya:   Minimax completo (tablero 3x3, búsqueda total)
   - Conecta Cuatro: Minimax + poda alpha-beta, profundidad 4,
                     heurística de ventanas deslizantes + preferencia centro
   - Buscaminas:     CSP (propagación de restricciones) para celdas seguras,
                     fallback a celda aleatoria no minada
   - Flujo: get_best_move() primero; si devuelve None → DeepSeek LLM
   - Badge "IA disponible" en los juegos que la tienen

4. SISTEMA DE SALAS
   - Crear sala → código aleatorio de 8 chars
   - Unirse por código (input en home)
   - Salas públicas listadas en home, poll cada 15s
   - Spectating: entrar a sala en curso, chat como [Espectador],
                 sin interacción con el juego
   - Room timeout: sala inactiva 10 min → evento room_closed → redirect a home

5. REMATCH
   - Botón post-game-over para jugadores humanos
   - Todos los humanos deben votar (progreso visible en UI)
   - Al completarse: sala vuelve a "waiting", game_state reseteado,
     nadie se desconecta

6. PRESENCIA ONLINE + AMIGOS + INVITACIONES
   - Sidebar en home: lista de usuarios online (excepto yo)
   - Botón ➕ añade amigo (guarda en Firestore)
   - Botón ✉️ abre picker de juego → crea sala → envía invite_received
     vía WS directo al destinatario
   - Toast de invitación con countdown 15s (Accept / Reject)
   - Presencia se re-registra en cada reconexión WS

7. PERFIL DE USUARIO
   - Pestaña Estadísticas: puntos, nivel, partidas por juego
   - Pestaña Juegos: stats por juego (victorias, derrotas, puntos)
   - Pestaña Amigos: carga lazy, muestra tarjetas con avatar, nombre,
     nivel, total_points, mejor juego y sus puntos
     (los UIDs de Firestore se resuelven a perfiles completos vía
     GET /api/users/friends)
   - Leaderboard global en sidebar de home

8. PANEL DE ADMINISTRACIÓN
   - Ruta protegida /admin
   - Gestión de usuarios y estadísticas globales


PONG — DETALLES DE IMPLEMENTACIÓN
-----------------------------------

Constantes del backend (backend/app/games/pong/game.py):
  PADDLE_SPEED      = 8
  INITIAL_SPEED     = 10.0
  MAX_SPEED         = 20.0
  SPEED_INCREMENT   = 0.5
  WALL_SPEED_INCREMENT = 0.3   ← incremento de velocidad al chocar con pared
  SERVE_TICKS       = 20
  BALL_SIZE         = 20

Física de paredes (choques arriba/abajo):
  - Cada choque aumenta la velocidad en WALL_SPEED_INCREMENT (máx MAX_SPEED)
  - La componente vy se aplana un 20% (vy_mag = abs(vy) * 0.80)
  - vx se recalcula para mantener la nueva velocidad resultante
  - Efecto: la bola va progresivamente más "recta" horizontalmente

Configuración de win_score en lobby:
  - El líder puede elegir 5, 10 o 15 puntos para ganar
  - Evento WS: pong_set_win_score → valida y hace broadcast de room_state
  - room["pong_win_score"] almacena la elección
  - _start_game pasa win_score a get_initial_state()

Diseño visual del canvas (pong.component.ts):
  - Sin línea central
  - Sin neon/glow en paletas (roundRect simple)
  - Bola: círculo SVG-equivalente (ctx.arc + clip), radio = BALL_SIZE/2 = 10px
  - Rainbow ball: radial gradient HSL cuando speed >= RAINBOW_THRESHOLD (13)
  - Puntuaciones: grandes (200px Impact) en cada mitad del tablero
    · Flash a alpha=0.82 durante 600ms al marcar un punto
    · Fade hasta alpha=0.05 en 1600ms (apenas visibles)
  - Sin texto "READY", sin "/15"

SVG de assets (frontend/public/pong/):
  - Ball.svg  (1 archivo — geometría de círculo, implementada como arc nativo)


SNAKE — DETALLES DE IMPLEMENTACIÓN
-------------------------------------

Constantes del frontend (snake.component.ts):
  CELL  = 64   ← tamaño de celda en píxeles (canvas interno: 64×24 = 1536px)
  BOARD = 24   ← celdas por lado

  Nota: CELL fue ajustado manualmente a 64 tras varios incrementos (20→28→32→64).
  La visualización en pantalla se escala con CSS: min(672px, calc(100vw - 64px)).
  El CELL actual puede ser excesivo; si los sprites se ven demasiado grandes
  comparados con el tablero o generan problemas, considerar bajarlo a 32 o 40.

Sprites SVG (frontend/public/serpiente/) — 9 archivos en total:
  HEAD.svg           ← cabeza del jugador propio (cara mirando ARRIBA)
  HEAD_2.svg         ← cabeza de rivales
  BODY.svg           ← segmento recto vertical
  BODY_UP_RIGHT.svg  ← esquina que conecta ARRIBA y DERECHA
  BODY_DOWN_RIGHT.svg← esquina que conecta ABAJO y DERECHA
  BODY_DOWN_LEFT.svg ← esquina que conecta ABAJO y IZQUIERDA
  BODY_UP_LEFT.svg   ← esquina que conecta ARRIBA y IZQUIERDA
  TAIL.svg           ← cola (punta ABAJO, conexión ARRIBA)
  snakePoints.svg    ← icono decorativo (UI de puntuación)

Orientaciones canónicas de los SVGs:
  HEAD / HEAD_2  → cara mirando ARRIBA (rotar para otras direcciones)
  TAIL           → punta ABAJO, conexión al cuerpo en la PARTE SUPERIOR
  BODY           → segmento vertical (arriba ↔ abajo)
  BODY_*         → esquinas sin rotación adicional (nombre indica las conexiones)

Sistema de tintado por jugador:
  - Todos los SVGs tienen fill="white"
  - globalCompositeOperation = 'source-in' aplica el color del jugador
  - Cada combinación (color, sprite, angleDeg) se cachea como bitmap CELL×CELL
  - Coste por segmento por frame: 1 drawImage() (antes: ~6 ops canvas costosas)

Optimización de rendimiento (solución al FPS drop con serpientes largas):
  - Pre-renderizado del fondo estático (grid) una sola vez en bgCanvas
  - Cache de sprites tintados: Map<"color:sprite:angle", HTMLCanvasElement>
  - Invalidación: al cambiar colores → clear(); al recargar SVG → borrar claves
  - Fondo: 1 drawImage() por frame en lugar de N² líneas de grid

Estado actual / pendiente en Snake:
  - Los sprites se ven correctamente tintados con los colores de cada jugador
  - La rotación de HEAD y TAIL funciona según la dirección actual
  - Las esquinas del body usan sprites específicos sin rotación adicional
  - PROBLEMA CONOCIDO: el ajuste visual no queda del todo bien en algunos casos;
    el valor de CELL (actualmente 64) fue modificado manualmente y puede que no
    sea el óptimo. La relación entre el tamaño del canvas interno y el CSS de
    visualización necesita revisión para que los sprites encajen correctamente
    en las celdas del tablero.
  - Pendiente: revisar y ajustar la escala/tamaño de los sprites SVG dentro
    de cada celda para que haya un padding natural y no queden pegados al borde.

BUGS ENCONTRADOS Y SOLUCIONADOS
---------------------------------

1. RUTA FastAPI capturada por parámetro dinámico
   Problema:  GET /{game_id} declarado antes que GET /rooms
              → "/rooms" se interpretaba como game_id="rooms"
   Solución:  Declarar rutas literales primero (/rooms, /rooms/{room_id})
              y /{game_id} al final.

2. SPA 404 al recargar en Render
   Problema:  Render sirve archivos estáticos; rutas Angular como
              /profile no existen como archivos → 404.
   Solución:  frontend/public/_redirects con "/* /index.html 200"
              (Render da prioridad a este archivo sobre render.yaml routes).

3. Amigos mostraban UID en bruto
   Problema:  profile().friends era string[] de UIDs de Firestore.
   Solución:  Nuevo endpoint GET /api/users/friends que resuelve
              UIDs → perfiles completos (nombre, nivel, etc.).

4. Salas fantasma ("ghost rooms")
   Problema:  Sala en estado "playing" con ambos jugadores desconectados
              nunca pasaba a "finished" (no había más moves), quedaba
              en memoria indefinidamente.
   Solución:
     a) _auto_close_if_empty(): llamado al final de cada leave_room.
        - waiting/finished + sin conexiones → borrada inmediatamente.
        - playing + sin conexiones → last_activity adelantado para
          que el cleanup la cierre en ~60s (ventana de gracia).
     b) cleanup_stale_rooms(): también cierra salas "playing" con
        cero conexiones activas tras 2 min de gracia.

5. Carga lenta de salas en home
   Problema:  Promise.allSettled([rooms, users]) esperaba ambas
              respuestas; getOnlineUsers requiere auth token (más lento).
   Solución:  Dos .then() independientes: salas se muestran en cuanto
              llegan; usuarios online se actualizan por separado.
              Añadido indicador "Cargando salas..." (loadingRooms signal).

6. Fix Buscaminas: explosión de mina
   Problema:  Al explotar una mina, el winner se asignaba al jugador
              que pisó la mina en lugar del contrario.
   Solución:  winner = next_turn (el OTRO jugador gana), +50 pts.

7. WS gestionado por sala (bug de ciclo de vida)
   Problema:  GameRoom hacía ws.disconnect() en ngOnDestroy, cortando
              el WS global al salir de la sala → invitaciones dejaban
              de funcionar.
   Solución:  WS gestionado solo en App component. GameRoom solo envía
              "leave_room" en ngOnDestroy, nunca desconecta.

8. FPS drop en Snake al crecer la serpiente
   Problema:  Por cada segmento por frame se ejecutaban ~6 operaciones
              canvas costosas (SVG drawImage + composite source-in + fillRect).
              Con serpientes de 30+ segmentos el juego iba lento.
   Solución:  Cache de bitmaps pre-tintados (ver sección Snake arriba) +
              pre-renderizado del fondo estático en canvas separado.

9. Pong - bola chocando en paredes iba demasiado "diagonal"
   Problema:  Tras cada rebote la trayectoria mantenía mucha componente
              vertical, haciendo el juego aburrido y predecible.
   Solución:  Cada choque con pared aplana vy un 20% y aumenta velocidad;
              vx se recalcula para mantener la velocidad total, resultando
              en una trayectoria más horizontal con el tiempo.

INICIAR EL ENTORNO LOCAL
--------------------------

Windows (CMD/PowerShell):
  deploy.bat   ← abre dos ventanas: backend y frontend

Git Bash / Linux / Mac (deploy.sh):
  bash deploy.sh   ← equivalente al .bat, para terminales POSIX

El script:
  1. Comprueba que .env existe
  2. Crea backend/.venv si no existe e instala dependencias pip
  3. Instala dependencias npm en frontend/
  4. Lanza backend  → http://localhost:8000  (uvicorn --reload)
  5. Lanza frontend → http://localhost:4200  (ng serve)
  6. API Docs       → http://localhost:8000/docs


DEPLOY EN RENDER
-----------------
- Frontend: Static Site, build command: cd frontend && npm install && npm run build
  publish dir: frontend/dist/frontend/browser
  _redirects: frontend/public/_redirects ("/* /index.html 200")
- Backend:  Web Service, start: uvicorn main:app --host 0.0.0.0 --port $PORT
  root dir: backend/
- Variables de entorno en Render: DEEPSEEK_API_KEY, FIREBASE_* credentials
